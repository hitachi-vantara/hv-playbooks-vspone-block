# Playbook for GAD Compliance Report with Configurable Thresholds

## Overview
This Ansible playbook was customized to align with the environment's specific requirements. This modified playbook is executed to automate the generation of a comprehensive GAD compliance report, ensuring consistent validation of active-active volume replication health across connected storage systems. All compliance thresholds are externalized into a configurable variables file (`compliance_vars.yml`), allowing teams to adjust expected values without modifying the playbook logic.

The compliance report is generated by executing the following tasks:

**Retrieve GAD Pair Facts:** Using the hv_gad_facts module to gather all GAD pairs from the primary storage system. The playbook will fail if no GAD pairs are found.

**Retrieve Parity Group Facts:** Using the hv_paritygroup_facts module to gather all parity groups from the primary storage system for association with primary LDEVs.

**Retrieve Storage Pool Facts:** Using the hv_storagepool_facts module to gather all storage pools from the primary storage system for capacity and status compliance checks.

**Gather Primary LDEV Details:** Using the hv_ldev_facts module it loops over each GAD pair to retrieve detailed LDEV information for every primary volume from the primary storage system.

**Gather Secondary LDEV Details:** Using the hv_ldev_facts module it loops over each GAD pair to retrieve detailed LDEV information for every secondary volume from the secondary storage system.

**Build Final Consolidated Compliance Report:** A comprehensive compliance report is generated that evaluates GAD pairs, primary and secondary LDEVs, parity group associations, and storage pools against the configurable thresholds defined in `compliance_vars.yml`. Each check returns a PASSED or FAILED result.

**Write Final Compliance Report to File:** The generated compliance report is written to `/tmp/final_compliance_report.txt` for review and record keeping.

## Compliance Checks Performed
The report evaluates the following against configurable thresholds:

**GAD Pair Checks:** Copy pace track size and copy rate are validated against expected values.

**LDEV Checks (Primary and Secondary):** LDEV status, deduplication compression status, deduplication compression mode, path count, and provision type are validated against expected values.

**Capacity Match Check:** Primary and secondary LDEV capacities are compared to ensure they match.

**Parity Group Association:** Each primary LDEV is mapped to its associated parity group displaying RAID level, drive type, total capacity, and free capacity.

**Storage Pool Checks:** Pool status, available capacity, and used capacity rate are validated against expected thresholds.

## Configurable Thresholds (compliance_vars.yml)
All compliance thresholds are defined in `compliance_vars.yml` located in the same directory as the playbook. Below are the configurable variables and their default values:

| Variable | Default Value | Description |
|---|---|---|
| expected_gad_copy_pace | MEDIUM | Expected GAD copy pace track size |
| expected_gad_copy_rate | MEDIUM | Expected GAD copy rate |
| expected_ldev_status | NML | Expected LDEV status |
| expected_ldev_dedup_compression_status | ENABLED | Expected deduplication compression status |
| expected_ldev_dedup_compression_mode | compression_deduplication | Expected deduplication compression mode |
| expected_ldev_min_path_count | 2 | Minimum number of paths expected per LDEV |
| expected_ldev_provision_type | CVS | Expected provision type (substring match) |
| expected_pool_status | NORMAL | Expected storage pool status |
| pool_min_free_capacity_mb | 1024000 | Minimum available pool capacity in MB |
| pool_max_used_capacity_rate | 80 | Maximum allowed used capacity rate percentage |

To adjust thresholds, edit `compliance_vars.yml` directly. No changes to the playbook are required.
